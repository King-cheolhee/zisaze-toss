## 0. 실행 에이전트 필독 규칙 (모든 Phase에 우선)

1. **웹 무중단이 최우선.** rich1(기존 웹) 변경은 이 문서 §5 Phase 1에 명시된 **추가(additive) 작업만** 허용한다. 기존 라우트·컴포넌트·테이블 컬럼의 수정/삭제 금지. 명시되지 않은 rich1 변경이 필요해지면 **작업을 멈추고 사용자에게 보고**한다.
2. rich1을 건드리기 전에 `rich1/docs/AI_ONBOARDING.md`를 먼저 읽는다(rich1 CLAUDE.md 규칙). rich1 커밋은 dev 브랜치 → 검증 → main. 커밋 후 push.
3. 미니앱(신규 폴더 `zisaze-toss/`)은 자유롭게 개발하되, §3 토스 하드 제약을 위반하는 코드는 절대 작성하지 않는다. 코드 수정 후마다 §3 셀프체크를 돌린다.
4. **토스 콘솔의 "검토 요청"·"출시하기" 버튼은 절대 에이전트가 누르거나 누르라고 재촉하지 않는다.** 사용자가 직접 결정한다(출시는 즉시 100% 반영·롤백도 즉시라 비가역성 큼).
5. 비밀(API 키·토큰)은 `.env`에만. 코드에 하드코딩 금지.
6. 토스 공식 문서는 수시로 바뀐다. SDK 함수명·설정이 이 문서와 다르면 **공식 문서(developers-apps-in-toss.toss.im)가 정본**이다. 대부분 페이지는 URL 끝을 `.md`로 바꾸면 마크다운으로 읽힌다(전체 인덱스: `/llms.txt`).
7. 작업이 일단락되면 `zisaze-toss/zisaze-toss_session_v(n).md` 세션문서를 남긴다(글로벌 지침 형식).

## 2. 아키텍처 결정 (ADR)

### A1. 미니앱은 신규 독립 프로젝트 `c:\()안티그래비티\zisaze-toss\`
- 스택: **Vite + React 19 + TypeScript + `@apps-in-toss/web-framework`(SDK 2.4.5 이상)**. 순수 CSR.
- 산출물 `.ait` 번들을 토스 콘솔에 업로드하면 **토스 CDN이 호스팅**한다(자체 서버에서 프론트를 서빙하지 않음). 기존 URL 연결 방식은 앱인토스에 존재하지 않음(공식 튜토리얼 확인).
- 부트스트랩은 공식 스캐폴드(`npx create-ait-app@latest` → WebView 템플릿 선택) 우선. 실패 시 튜토리얼(tutorials/webview) 수동 셋업.
- 별도 git 저장소로 관리(비공개 `King-cheolhee/zisaze-toss` 권장). rich1 저장소에 섞지 않는다.
- 이유: rich1은 SSR(ISR)+API 라우트 62개로 서버 실행이 필수라 토스에 그대로 못 올림(SSR은 검수 금지 항목이기도 함). 별도 앱이 웹 무중단 원칙에도 부합.

### A2. 백엔드는 기존 rich1 API를 크로스오리진 호출
- 미니앱 → `https://www.zisaze.com/api/*` 직접 호출.
- **핵심 근거(조사 완료)**: rich1의 사용자 API는 전부 `Authorization: Bearer <JWT>` 헤더 인증이며 쿠키를 쓰지 않는다 → iOS 서드파티 쿠키 차단과 무관하게 그대로 재사용 가능.
- 필요한 rich1 변경은 CORS 허용뿐(§5 P1-1).

### A3. 사용자 식별 = `getAnonymousKey` (토스 로그인 없이)
- 비게임 미니앱 전용 API. **동의 화면·콘솔 설정·서버(mTLS) 구축 전부 불필요**. SDK 2.4.5+.
- 반환: `{ type: 'HASH', hash: string }` — 해당 미니앱에서만 유효한 사용자 고유 식별자.
- 흐름: 미니앱 최초 진입 → `getAnonymousKey()` → rich1 신규 엔드포인트 `POST /api/toss/session`에 hash 전달 → 서버가 users 행을 찾거나 생성 → **기존과 동일한 JWT 발급** → 이후 모든 기존 사용자 API(추천·북마크·프로필)가 무수정으로 동작.
- 토스 로그인(`appLogin`)을 안 쓰는 이유: mTLS 인증서 발급·서버 연동·동의 화면·약관 URL 등 부대작업이 크고, 지사제 미니앱에 필요한 건 "안정적 식별자"뿐. 정책상으로도 문제없음(로그인을 제공하려면 토스 로그인만 가능하다는 규정이며, 로그인 없는 식별키 방식은 비게임 표준 패턴). 업그레이드 경로는 Phase 5.
- ⚠️ 문서에 "기기 변경 시 동일 hash 유지" 여부가 명시되어 있지 않음 → **Phase 3 샌드박스/실기기에서 최우선 검증**(§7 R1). 샌드박스는 mock 값을 반환하므로 실기기 확인 필수.

### A4. rich1 변경은 additive 3종 + 검증 1종
1. `src/middleware.ts`에 토스미니 도메인 CORS 허용 추가
2. `users` 테이블에 `toss_user_key` 컬럼 추가(ADD COLUMN, nullable, UNIQUE)
3. `POST /api/toss/session` 라우트 신설
4. (코드 수정 아님) 빈 프로필 사용자에 대한 추천 동작 검증(P1-5)
→ 기존 웹 사용자·관리자 흐름에는 코드 경로가 전혀 겹치지 않는다.

### A5. 외부 이동은 전부 `openURL()`
- 원문보기(정부기관 공고), 개인정보처리방침 등 외부 URL은 `window.open`/`target="_blank"` 대신 SDK `openURL(url)` 사용(WebView에선 브라우저 탭, 기본 앱 전환).
- 정부 원문 이동은 토스 허용 예외("공공기관 공식 페이지 연결", "단순 정보 확인용 이동")에 해당 — 검수 메모에 명기(§5 P3-4).

### A6. 웹의 금지 요소는 미니앱에 "미포함"으로 처리
- GA(gtag)·카카오 공유 SDK·웹푸시(sw.js)·ad_banners·자체 로그인 화면 → 새 앱이므로 처음부터 넣지 않는다. 분석이 필요하면 `@apps-in-toss/web-analytics`만 사용.

### A7. 선호 분야 온보딩 = 추천의 1급 재료 (v2 신설 — 핵심)

**요구**: 회원가입(사업자번호·주소·업종 등)이 없는 토스 사용자는 추천 재료가 빈약하다. 최초 실행 시 **선호 분야를 칩 방식으로 선택**받고(예: AI, 청년창업, 인건비), 설정에서 언제든 수정 가능하게 하여 "완전하지 않지만 간접적인 추천"을 성립시킨다.

**설계 (기존 파이프라인 100% 재사용 — 신규 추천 로직 없음)**:
1. 온보딩에서 선택한 칩들(+선택적 자유입력 1줄)을 합성해 `preference_text`로 만들어 기존 `PUT /api/users/me`로 저장한다(예: 칩 [AI, 청년창업, 인건비] → "AI, 청년창업, 인건비(고용지원) 분야의 지원사업을 선호합니다").
2. 서버의 **기존 선호글 파이프라인**이 그대로 작동한다: preference_text 저장 시 Gemini 1회 분류(`preference-classify.ts`) → `preference_signals.priority_keywords`(우선순위 동의어 그룹) 생성 → `/api/recommend`가 가점(`preference-boost.ts`, RANK_WEIGHTS [40,24,14,8]×위치가중, 상한 50)을 ruleScore에 합산해 순위 결정. `PREF_BONUS_ENABLED` 토글로 보호되는 검증된 라이브 기능이다.
3. **빈 프로필에서도 추천이 성립하는 근거(코드 분석 완료, `src/lib/recommend-matcher.ts`)**: 매칭 엔진의 필수 축은 지역·업종 2개인데, ①지역 — 사용자 지역이 없으면 **전국 대상 사업은 통과**(+5) ②업종 — 사용자 업종이 없으면 **'전 업종' 사업은 통과**(+10). 즉 빈 프로필 사용자의 추천 풀 = "전국×전업종" 사업 집합이 되고, 그 안에서 선호 키워드 가점(최대 +50)이 기본점(약 30~40점대)을 지배하므로 **선택한 분야의 사업이 실제로 최상단에 오른다**. 코드 수정 없이 성립.
4. **품질 레버**: 온보딩에서 ①선호 칩(핵심) 다음에 ②지역(시/도 1탭, 선택)을 받으면 지역 사업까지 풀에 들어와 추천 폭이 크게 넓어진다(지역 매칭 +25~30점). ③업종·규모는 "나중에 설정"으로 미뤄 온보딩을 가볍게 유지한다.
5. **한계 명시(사용자 기대치 관리)**: 가점은 자격 통과(matched) 사업의 순위만 조정한다. 빈 프로필이면 지역 한정·특정 업종 한정 사업은 풀에서 빠진다 — 이것이 "간접 추천"의 정확한 의미다. 지역·업종을 채울수록 정확해진다는 안내 문구를 추천 영역에 상시 노출한다.
6. **AI 고지 연계(H16)**: 선호 분류에 Gemini(생성형 AI)를 1회 사용하므로, 온보딩 선호 입력 화면에 "입력하신 선호는 AI가 분석해 추천에 활용해요" 고지를 포함한다(별도 팝업 아닌 인라인 문구 — 다크패턴 H9 회피).

**선호 칩 후보 (초안 — P2-3에서 실데이터로 확정, §11 O6)**:
`AI·디지털전환` `청년창업` `인건비·고용지원` `수출·해외판로` `R&D·기술개발` `마케팅·판로개척` `시설·설비투자` `소상공인 경영안정` `제조·스마트공장` `창업 초기지원` `정책자금·융자` `교육·컨설팅` `여성기업` `관광·콘텐츠`
- 확정 절차: programs 테이블의 category 분포·ai_target_industries 상위 값·자주 등장하는 지원 유형을 집계해 12~16개로 조정(실데이터에 없는 칩은 가점이 안 붙어 무의미하므로 반드시 대조). 다중 선택(권장 1~5개) + "직접 입력" 1칸(선택).

## 4. 시스템 구성

```
[토스 앱 (Android 7+/iOS 16+, 만 19세+)]
   └─ 지사제 미니앱 (.ait 번들, 토스 CDN 호스팅, WebView/CSR)
        │  getAnonymousKey() → hash
        │  fetch https://www.zisaze.com/api/*  (Authorization: Bearer JWT)
        ▼
[기존 rich1 서버 — AWS Lightsail + Nginx + PM2 + Cloudflare]  ← 변경 3종만(additive)
   ├─ (신설) POST /api/toss/session   : hash → users upsert → JWT 발급
   ├─ (수정) src/middleware.ts        : 토스미니 2개 오리진 CORS 허용
   └─ (기존 재사용) /api/programs, /api/recommend, /api/users/me(선호 분류 포함), /api/bookmarks, /api/hidden …
        ▼
[Supabase PostgreSQL]  ← users에 toss_user_key 컬럼만 추가
```

미니앱이 호출하는 API 계약(전부 기존, 신설 1개):

| API | 용도 | 인증 |
|---|---|---|
| `POST /api/toss/session` **(신설)** | hash → `{ success, user, token }` (기존 login 응답과 동일 형태) | 없음(hash가 자격) |
| `GET /api/programs?fields=card&…` | 목록/검색/필터 (경량 필드) | 불필요 |
| `GET /api/programs/[id]` | 상세(AI 요약 포함) | 불필요 |
| `GET /api/recommend` | 맞춤 추천(선호 가점 포함 — 기존 엔진 그대로) | Bearer |
| `PUT /api/users/me` | **선호(preference_text)**·지역·업종 등 프로필 저장 → 서버가 Gemini 분류로 priority_keywords 갱신 | Bearer |
| `GET /api/users/me` | 프로필·선호 조회(설정 화면 초기값) | Bearer |
| `GET/POST/DELETE /api/bookmarks`, `GET /api/bookmarks/ids` | 북마크 | Bearer |
| `GET/POST /api/hidden` | 관심없음(선택 기능) | Bearer |
| `DELETE /api/toss/session` 또는 기존 탈퇴 API | 데이터 삭제 | Bearer |

> 구현 전 각 라우트의 실제 요청/응답 스키마를 코드에서 확인할 것(온보딩 문서·`rich1/src/app/api/**`). 특히 `PUT /api/users/me`의 preference_text 전송 규약(dirty 플래그 — 재분류 트리거 조건)을 웹 `EditProfileModal.tsx` 구현과 동일하게 맞출 것. 응답 구조를 임의로 바꾸지 말 것(웹 프론트와 공유).

## 6. 데이터 설계

- `users.toss_user_key text UNIQUE NULL` — 미니앱 사용자만 값 보유. 기존 웹 사용자는 NULL(영향 없음).
- 미니앱 사용자 행: `email`은 스키마가 NOT NULL이면 합성 이메일(`toss_<hash12>@miniapp.zisaze.com`), `password_hash`는 임의 난수(로그인 불가 값), 프로필 필드는 온보딩 입력값(전부 선택).
- **선호 데이터는 기존 구조 그대로**: `users.preference_text`(칩 합성 문장 + 자유입력), `users.preference_signals.priority_keywords`(서버 Gemini 분류 산출). 추천 가점도 기존 `preference-boost` 그대로.
- 기존 테이블 재사용: `user_bookmarks`, `user_hidden_programs` — user_id FK 기반이라 무수정 동작 예상(구현 시 각 라우트에서 확인).
- 삭제(탈퇴): 해당 user 행 + 연관 행(bookmarks/hidden 등) 삭제. 개인정보보호법 "지체 없이 파기" 충족.
- 웹 계정 ↔ 미니앱 계정 연동은 하지 않음(YAGNI — 필요해지면 Phase 5에서 설계).
